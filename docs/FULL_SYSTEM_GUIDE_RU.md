# AssetChain — полное описание сайта и логики работы

Документ описывает текущее поведение системы по исходному коду: роли пользователей, страницы и кнопки, поступление товаров, перемещения между складами, согласование, Telegram-бот, импорт Excel, API и связи таблиц базы данных.

## 1. Назначение системы

AssetChain — система количественного складского учёта. Она хранит:

- пользователей и их роли;
- склады, ответственных и согласующих;
- единую номенклатуру товаров;
- фактический остаток каждого товара на каждом складе;
- заявки на перемещение товаров между складами;
- историю всех поступлений и этапов перемещений.

Главный принцип учёта:

- обычное поступление сразу увеличивает остаток на выбранном складе;
- создание заявки на перемещение остаток не меняет;
- согласование заявки остаток не меняет;
- подтверждение отправки уменьшает остаток склада-отправителя;
- подтверждение получения увеличивает остаток склада-получателя.

## 2. Роли и права

### 2.1. Главный администратор

Главным администратором считается:

- пользователь с ролью `Главный администратор`; или
- Django-суперпользователь.

Он может:

- видеть все склады, остатки, перемещения и операции;
- создавать новую номенклатуру;
- оформлять поступление на любой активный склад;
- создавать перемещение с любого активного склада;
- выполнять любое согласование, отклонение, отправку и получение;
- импортировать данные из Excel;
- создавать и редактировать пользователей;
- создавать и редактировать склады;
- открывать техническую Django Admin.

### 2.2. Ответственный за склад

Ответственный назначается конкретному складу.

Он может:

- видеть склад, за который отвечает;
- видеть перемещения, связанные с его складом;
- видеть доступную ему историю операций;
- оформлять поступление на свой склад;
- создавать перемещение со своего склада;
- подтверждать отправку со своего склада;
- подтверждать получение на свой склад.

Ответственный не может согласовывать заявку, если отдельно не назначен согласующим. Роль пользователя сама по себе не даёт доступ ко всем складам: важна привязка пользователя к полю `Ответственный` конкретного склада.

### 2.3. Согласующий

Согласующий назначается конкретному складу-отправителю.

Он может:

- видеть назначенный ему склад;
- видеть перемещения, отправляемые с этого склада;
- согласовать заявку;
- отклонить заявку.

Согласующий не списывает товар и не подтверждает получение.

### 2.4. Неактивный пользователь

Если у пользователя выключен флаг `Активен`, он больше не может войти в систему. Пользователя рекомендуется отключать, а не удалять: тогда ссылки на него в истории операций сохраняются.

## 3. Какие данные видит пользователь

Главный администратор видит всё.

Обычный пользователь видит склады, где он:

- ответственный; или
- согласующий.

Обычный пользователь видит перемещение, если выполняется хотя бы одно условие:

- он ответственный склада-отправителя;
- он ответственный склада-получателя;
- он согласующий склада-отправителя;
- он сам создал перемещение.

История операций показывается по тем же доступным перемещениям и складам.

Номенклатура доступна всем вошедшим пользователям. В текущей реализации на странице номенклатуры также выводятся остатки товара по всем складам, даже если отдельная страница `Остатки` ограничивает список доступными пользователю складами.

## 4. Первый запуск и первоначальная настройка

Рекомендуемый порядок настройки:

1. Создать первого суперпользователя командой `python manage.py createsuperuser`.
2. Войти на сайт.
3. Открыть `Управление`.
4. Создать пользователей с ролями `Ответственный за склад` и `Согласующий`.
5. Создать склады.
6. Для каждого склада выбрать ответственного.
7. При необходимости выбрать согласующего.
8. Создать товары вручную или импортировать данные из Excel.
9. При использовании Telegram передать каждому пользователю его персональную ссылку подключения.

Для каждого активного склада обязательно назначается согласующий. Без него создать перемещение нельзя.

## 5. Вход, общее меню и служебные элементы

### Страница входа

Поля:

- `Логин` — имя пользователя;
- `Пароль` — пароль пользователя.

Кнопка `Войти` проверяет данные. При ошибке показывается сообщение `Неверный логин или пароль`.

После успешного входа пользователь попадает на страницу `Обзор`.

### Левое меню

- `AssetChain` или логотип `A` — переход на обзор.
- `Обзор` — главные показатели и последние действия.
- `Номенклатура` — список моделей товаров.
- `Остатки` — количество товаров по доступным складам.
- `Перемещения` — заявки на передачу товаров.
- `История` — журнал операций.
- `Импорт Excel` — массовая загрузка; виден только главному администратору.
- `Управление` — пользователи и склады; виден только главному администратору.
- Кнопка со стрелкой выхода возле пользователя — завершение сеанса.

На телефоне:

- кнопка `☰` открывает боковое меню;
- `×`, затемнённая область за меню или выбор пункта закрывают/сменяют экран;
- надпись `AssetChain` в верхней панели ведёт на обзор.

Формы защищены от случайной повторной отправки: после нажатия кнопка сохранения временно блокируется и показывает `Сохраняем…`.

## 6. Страница «Обзор»

### Верхние кнопки

- `+ Поступление` — открыть форму приёмки существующего товара на склад.
- `Новое перемещение` — создать заявку на передачу товара между складами.

### Показатели

- `Склады` — количество складов, доступных текущему пользователю.
- `Модели товаров` — общее количество позиций номенклатуры.
- `Единиц на складах` — сумма остатков только по доступным пользователю складам.
- `Перемещения` — количество доступных пользователю перемещений за всё время.
- `Сейчас в пути` — количество доступных перемещений со статусом `В пути`.

Карточки являются информационными и не выполняют действие при нажатии.

### Последние операции

Показываются 10 последних доступных записей:

- дата;
- название операции;
- товар;
- пользователь, выполнивший действие.

Кнопка `Вся история` открывает полный журнал операций.

### Низкие остатки

Показываются до 10 позиций, где остаток равен 3 или меньше:

- 0 выделяется красным;
- 1–3 выделяются жёлтым.

## 7. Номенклатура

Номенклатура — единый справочник моделей товаров. Один товар может одновременно иметь остатки на нескольких складах.

### Поиск и фильтры

- Поле `Название, артикул или штрихкод` ищет по любому из этих значений.
- Список `Все категории` ограничивает товары выбранной категорией.
- `Найти` применяет поиск и фильтр.
- `Сбросить` очищает фильтры.

USB- или Bluetooth-сканер обычно вводит штрихкод как клавиатура. Нужно поставить курсор в строку поиска и считать код.

### Таблица

Для товара показываются:

- название;
- описание, если оно заполнено;
- категория;
- пятизначный артикул;
- штрихкод;
- остатки по складам;
- кнопка печати этикетки.

### Кнопка «+ Добавить товар»

Видна только главному администратору. Открывает форму `Новая номенклатура`.

Поля:

- `Наименование` — обязательное название модели;
- `Категория` — обязательная категория;
- `Описание` — необязательное пояснение;
- `Склад` — активный склад первого поступления;
- `Начальное количество` — от 1 до 1 000 000.

Кнопки:

- `Сохранить` создаёт товар и его первый остаток;
- `Отмена` возвращает на предыдущую страницу.

При сохранении система автоматически:

1. Генерирует свободный пятизначный артикул от `00001` до `99999`.
2. Генерирует уникальный внутренний EAN-13 со стартовыми цифрами `21`.
3. Создаёт строку товара в номенклатуре.
4. Создаёт остаток на выбранном складе.
5. Записывает в историю операцию `Первичное поступление`.

### Кнопка «Печать этикетки»

Открывает этикетку формата 58×40 мм.

На этикетке:

- название товара;
- количество;
- артикул;
- штрихкод;
- число штрихкода текстом.

Поле количества принимает значение от 1 до 9999.

- `Обновить` меняет напечатанное число на этикетке.
- `Печать` открывает стандартное окно печати браузера.

Важно: изменение количества на этикетке не изменяет остаток в базе. Это только текст для печати.

## 8. Что делать, когда поступил товар

Сначала определить, есть ли модель в номенклатуре.

### 8.1. Поступила уже существующая модель

1. Открыть `Остатки`.
2. Нажать `+ Поступление товара`.
3. Выбрать склад поступления.
4. Выбрать товар.
5. Указать фактически принятое количество.
6. Нажать `Сохранить`.

После сохранения:

- если строки остатка ещё не было, она создаётся;
- указанное количество прибавляется к текущему остатку;
- создаётся запись `Поступление товара`;
- пользователь возвращается на страницу остатков;
- показывается новый итоговый остаток.

Обычный пользователь может выбрать только активный склад, где он назначен ответственным. Главный администратор может выбрать любой активный склад.

### 8.2. Поступила новая модель, которой ещё нет в справочнике

Это делает главный администратор прямо в общей форме поступления:

1. Открыть `Остатки`.
2. Нажать `+ Поступление товара`.
3. Убедиться, что товара нет среди существующей номенклатуры.
4. Выбрать склад и указать количество.
5. В блоке новой номенклатуры заполнить общее название модели, категорию и описание.
6. Нажать `Оформить поступление`.

Система одновременно создаёт номенклатуру, общий артикул, общий штрихкод и начальный остаток. Если товар с таким названием уже существует без учёта регистра, создание блокируется и система предлагает выбрать существующую позицию.

### 8.3. Поступило много новых данных

Главный администратор может использовать `Импорт Excel`. Импорт не прибавляет переданные остатки к старым: строка листа `Остатки` устанавливает точное значение остатка для пары «склад + товар».

## 9. Страница «Остатки»

### Кнопка «+ Поступление товара»

Открывает форму поступления, описанную выше.

### Фильтры

- Строка поиска — товар, артикул или штрихкод.
- `Все склады` — выбрать склад.
- `Все категории` — выбрать категорию.
- `Показать` — применить фильтры.
- `×` — очистить все активные фильтры.

### Таблица

Показывает:

- склад;
- товар;
- категорию;
- артикул;
- количество;
- дату и время последнего изменения остатка.

Цвет количества:

- красный — 0;
- жёлтый — от 1 до 3;
- зелёный — больше 3.

Одна строка таблицы соответствует одной уникальной паре `склад + товар`.

## 10. Полный сценарий перемещения товара

### 10.1. Создание заявки

1. Открыть `Перемещения`.
2. Нажать `+ Новое перемещение`.
3. В поле `Откуда` выбрать склад-отправитель.
4. В поле `Куда` выбрать другой активный склад.
5. Выбрать товар.
6. Указать количество.
7. Нажать `Сохранить`.

Правила:

- склад отправителя и получателя должны отличаться;
- количество должно быть не меньше 1;
- на складе-отправителе должно хватать товара;
- обычный пользователь может отправлять только со склада, где он ответственный;
- главный администратор может отправлять с любого активного склада.

При создании заявки остаток не резервируется и не списывается.

`На согласовании → Ожидает отправки → В пути → Завершено`

Согласование обязательно для каждого перемещения.

### 10.2. Согласование

Согласующий или главный администратор:

1. Открывает `Перемещения`.
2. Находит заявку со статусом `На согласовании`.
3. Нажимает номер заявки `#...`.
4. Проверяет маршрут, товар и количество.
5. Нажимает `Согласовать` или `Отклонить`.

`Согласовать`:

- меняет статус на `Ожидает отправки`;
- записывает дату согласования;
- добавляет операцию `Перемещение согласовано`;
- уведомляет ответственного склада-отправителя через Telegram, если бот подключён.

`Отклонить`:

- меняет статус на `Отклонено`;
- добавляет операцию `Перемещение отклонено`;
- остатки не меняются;
- дальнейших действий по этой заявке нет.

### 10.3. Подтверждение отправки

Ответственный склада-отправителя или главный администратор:

1. Открывает заявку со статусом `Ожидает отправки`.
2. Нажимает `Подтвердить отправку`.

Система повторно проверяет остаток. Это важно, потому что при создании заявки товар не резервируется и мог быть использован другой операцией.

Если товара хватает:

- количество списывается со склада-отправителя;
- статус меняется на `В пути`;
- записывается время отправки;
- в историю добавляется `Товар отгружен`;
- ответственному склада-получателя отправляется Telegram-уведомление.

Если товара уже недостаточно, отправка блокируется, статус и остаток не меняются.

### 10.4. Подтверждение получения

Ответственный склада-получателя или главный администратор:

1. Фактически принимает товар.
2. Открывает заявку со статусом `В пути`.
3. Нажимает `Подтвердить получение`.

После этого:

- создаётся строка остатка на складе-получателе, если её ещё не было;
- количество прибавляется к остатку получателя;
- статус меняется на `Завершено`;
- записывается время получения;
- в историю добавляется `Товар получен, перемещение завершено`.

Повторно отправить или повторно получить одну заявку нельзя: действие разрешено только из строго определённого статуса.

## 11. Страница «Перемещения»

### Кнопка «+ Новое перемещение»

Открывает форму создания заявки.

### Фильтры

- Поиск — товар, артикул, штрихкод или название склада.
- `Все статусы` — фильтр по этапу заявки.
- `Все склады` — заявки, где склад является отправителем или получателем.
- `Показать` — применить.
- `×` — сбросить.

### Таблица

- `№` — номер заявки; нажатие открывает карточку.
- `Маршрут` — склад-отправитель → склад-получатель.
- `Товар` — название и артикул.
- `Количество`.
- `Статус`.
- `Создано` — дата и время создания.

## 12. Карточка перемещения

В карточке показываются:

- текущий статус;
- откуда;
- куда;
- товар;
- артикул;
- количество;
- кто создал;
- дата создания.

Кнопка `← К списку` возвращает к перечню перемещений.

Доступные кнопки зависят одновременно от статуса и прав пользователя:

- `Согласовать` — только при статусе `На согласовании`, для согласующего склада-отправителя или администратора;
- `Отклонить` — при тех же условиях;
- `Подтвердить отправку` — только при статусе `Ожидает отправки`, для ответственного склада-отправителя или администратора;
- `Подтвердить получение` — только при статусе `В пути`, для ответственного склада-получателя или администратора.

Блок `Хронология` показывает все этапы этой заявки, пользователя и время каждого действия.

## 13. Статусы перемещения

| Код в БД | Название | Значение |
|---|---|---|
| `created` | Создано | Предусмотрен моделью, но обычный интерфейс не оставляет новую заявку в этом статусе |
| `pending_approval` | На согласовании | Ожидается решение согласующего |
| `awaiting_shipment` | Ожидает отправки | Заявка разрешена, но товар ещё не списан |
| `in_transit` | В пути | Товар списан у отправителя, но ещё не зачислен получателю |
| `completed` | Завершено | Получатель подтвердил товар, остаток зачислен |
| `rejected` | Отклонено | Согласующий отказал, остатки не менялись |
| `cancelled` | Отменено | Статус есть в модели, но кнопки и бизнес-операции отмены сейчас не реализованы |

Схема переходов:

```text
Если есть согласующий:
На согласовании ──Согласовать──> Ожидает отправки ──Отправить──> В пути ──Получить──> Завершено
       │
       └──Отклонить──> Отклонено

```

## 14. История операций

История — журнал фактически выполненных действий. Через обычный интерфейс записи нельзя редактировать или удалять.

### Фильтры

- Поиск по товару, артикулу, названию действия, логину, имени или фамилии пользователя.
- `Все операции` — выбрать тип действия.
- `Все склады` — выбрать склад, участвующий как отправитель или получатель.
- `Показать` — применить.
- `×` — сбросить.

### Колонки

- дата;
- операция;
- товар и артикул;
- склад или маршрут;
- количество;
- пользователь;
- статус.

В журнал попадают:

- `Первичное поступление`;
- `Поступление товара`;
- `Создано перемещение`;
- `Перемещение согласовано`;
- `Перемещение отклонено`;
- `Товар отгружен`;
- `Товар получен, перемещение завершено`;
- `Импорт данных из Excel`.

Для перемещения журнал хранит снимок маршрута, товара, количества и статуса на момент операции.

## 15. Импорт Excel

Раздел доступен только главному администратору.

### Кнопки

- Выбор файла — прикрепить `.xlsx`.
- `Импортировать` — проверить и загрузить весь файл.

Нужны листы с точными названиями:

### Лист «Номенклатура»

Обязательные колонки:

- `Наименование`;
- `Артикул`;
- `Штрихкод`.

Необязательные:

- `Категория`;
- `Описание`.

Артикул должен состоять ровно из пяти цифр. Артикул и штрихкод должны быть уникальны.

Если артикул уже существует и штрихкод совпадает, обновляются название, категория и описание. Если тот же артикул пришёл с другим штрихкодом, импорт прерывается.

### Лист «Склады»

Обязательная колонка:

- `Наименование`.

Если склад уже существует, второй склад не создаётся. Новый склад создаётся без ответственного и согласующего — их нужно назначить в разделе `Управление`.

### Лист «Остатки»

Обязательные колонки:

- `Склад`;
- `Артикул`;
- `Количество`.

Количество должно быть целым неотрицательным числом. Значение устанавливается как точный текущий остаток, а не прибавляется к существующему.

Весь импорт выполняется одной транзакцией. Ошибка в любой строке отменяет все изменения файла.

После успешного импорта создаётся одна общая запись истории `Импорт данных из Excel`. Отдельные записи истории для каждой строки не создаются.

## 17. Управление складами

### Кнопка «+ Склад»

Поля:

- `Название` — уникальное название склада;
- `Ответственный` — активный пользователь с ролью ответственного или администратора;
- `Согласующий` — активный пользователь с ролью согласующего или администратора;
- `Активен`.

Ответственный и согласующий могут быть не назначены, но без ответственного обычному пользователю будет некому оформлять отправку или получение.

Один и тот же пользователь не может быть одновременно ответственным и согласующим одного склада.

Согласующий обязателен для активного склада. Без назначенного согласующего новые перемещения создать нельзя.

### Кнопка «Изменить»

Позволяет поменять название, ответственного, согласующего и активность.

Отключённый склад:

- сохраняется в базе и истории;
- не предлагается в формах нового поступления и перемещения;
- продолжает существовать для старых данных.

### Кнопка «Техническая Django Admin»

Открывает `/admin/`. Доступ туда дополнительно зависит от Django-флага `is_staff`, а не только от роли `Главный администратор`.

В Django Admin доступны технические таблицы пользователей, складов, товаров, остатков, перемещений и истории. История там настроена только для чтения.

## 18. Telegram-бот

Для работы нужны `TELEGRAM_BOT_TOKEN`, `TELEGRAM_BOT_USERNAME` и запущенная команда:

```bash
python manage.py runbot
```

### Подключение

1. Администратор открывает `Управление`.
2. Копирует пользователю `Ссылку подключения`.
3. Пользователь открывает её в своём Telegram.
4. Нажимает `Start`.
5. Бот связывает Telegram ID с учётной записью.

Один Telegram ID нельзя привязать к двум пользователям.

### Уведомления

После создания или перехода заявки бот направляет сообщение следующему исполнителю:

- `На согласовании` → согласующему склада-отправителя, кнопки `Согласовать` и `Отклонить`;
- `Ожидает отправки` → ответственному склада-отправителя, кнопка `Подтвердить отправку`;
- `В пути` → ответственному склада-получателя, кнопка `Подтвердить получение`.

В сообщении есть:

- номер перемещения;
- товар;
- количество;
- маршрут;
- статус.

После нажатия действие проходит те же проверки прав и статуса, что и на сайте. Затем кнопка убирается, а уведомление отправляется следующему исполнителю.

Все эти действия также остаются доступны вручную в карточке перемещения на сайте.

Главный администратор может нажать `Отвязать` возле подключённого Telegram в разделе `Управление`. После отвязки Telegram ID удаляется, а персональный токен подключения заменяется: старая ссылка больше не действует, для повторного подключения нужно использовать новую ссылку.

Если токен бота не задан, Telegram-уведомление просто пропускается. Ошибка отправки уведомления записывается в серверный лог и не отменяет складскую операцию.

## 19. REST API

Все API-запросы требуют активной Django-сессии пользователя.

### Склады

- `GET /api/warehouses/` — список доступных складов.
- `GET /api/warehouses/{id}/` — один доступный склад.

Только чтение.

### Товары

- `GET /api/products/` — вся номенклатура.
- `GET /api/products/?q=...` — поиск по названию, артикулу или штрихкоду.
- `GET /api/products/{id}/` — карточка товара.

Только чтение.

### Остатки

- `GET /api/stocks/` — остатки доступных складов.
- `GET /api/stocks/?warehouse={id}` — остатки одного склада.
- `GET /api/stocks/{id}/` — строка остатка.

Только чтение.

### Перемещения

- `GET /api/transfers/` — доступные перемещения.
- `GET /api/transfers/{id}/` — одно перемещение.
- `POST /api/transfers/` — создать перемещение.
- `POST /api/transfers/{id}/approve/` — согласовать.
- `POST /api/transfers/{id}/reject/` — отклонить.
- `POST /api/transfers/{id}/ship/` — подтвердить отправку.
- `POST /api/transfers/{id}/receive/` — подтвердить получение.

Создание принимает идентификаторы `source`, `destination`, `product` и число `quantity`.

API использует ту же складскую логику и проверки прав. После создания и изменения статуса через API также отправляются Telegram-уведомления.

## 20. База данных и связи сущностей

Основные таблицы приложения:

```text
User
 ├──< Warehouse.responsible
 ├──< Warehouse.approver
 ├──< Transfer.created_by
 └──< OperationLog.user

Warehouse
 ├──< Stock >── Product
 ├──< Transfer.source
 ├──< Transfer.destination
 ├──< OperationLog.source
 └──< OperationLog.destination

Transfer
 └──< OperationLog

Product
 ├──< Stock
 ├──< Transfer
 └──< OperationLog
```

Обозначение `A ──< B` означает: одной записи A соответствует много записей B.

### 20.1. User — пользователь

Основные поля:

- `id` — первичный ключ;
- `username` — уникальный логин;
- `password` — хеш пароля;
- `first_name`, `last_name`, `email`;
- `role` — `admin`, `responsible` или `approver`;
- `is_active` — разрешён ли вход;
- `is_staff`, `is_superuser` — технические права Django;
- `telegram_id` — уникальный ID Telegram;
- `telegram_link_token` — уникальный токен персональной ссылки.

Связи:

- один пользователь может отвечать за много складов;
- один пользователь может согласовывать много складов;
- один пользователь может создать много перемещений;
- один пользователь может выполнить много операций.

### 20.2. Warehouse — склад

Поля:

- `id`;
- `name` — уникальное название;
- `responsible_id` — ответственный пользователь, может быть пустым;
- `approver_id` — согласующий, может быть пустым;
- `is_active`;
- `created_at`.

Связи:

- склад имеет много строк остатков;
- склад участвует во многих исходящих перемещениях как `source`;
- склад участвует во многих входящих перемещениях как `destination`.

Удаление ответственного или согласующего запрещается защитой `PROTECT`, пока на пользователя ссылается склад.

### 20.3. Product — товар

Поля:

- `id`;
- `name`;
- `category`;
- `article` — уникальные 5 цифр;
- `barcode` — уникальный штрихкод;
- `description`;
- `created_at`;
- `imported` — был ли товар загружен из Excel.

Связи:

- товар может иметь много остатков на разных складах;
- товар может участвовать во многих перемещениях;
- товар может фигурировать во многих операциях.

### 20.4. Stock — остаток

Поля:

- `id`;
- `warehouse_id`;
- `product_id`;
- `quantity` — целое неотрицательное количество;
- `updated_at`.

Ограничение базы:

```text
UNIQUE (warehouse_id, product_id)
```

Поэтому для одного товара на одном складе существует не больше одной строки остатка.

`Stock` является таблицей связи «многие ко многим» между складами и товарами, но дополнительно хранит количество.

При удалении склада или товара связанные остатки удаляются каскадно. В обычной работе товары и склады, участвующие в перемещениях, удалить сложнее из-за защитных связей самих перемещений.

### 20.5. Transfer — перемещение

Поля:

- `id` — номер заявки;
- `source_id` — склад-отправитель;
- `destination_id` — склад-получатель;
- `product_id`;
- `quantity`;
- `status`;
- `created_by_id`;
- `created_at`;
- `approved_at`;
- `shipped_at`;
- `received_at`.

Склад-отправитель, склад-получатель, товар и создатель защищены через `PROTECT`: их нельзя удалить, пока на них ссылается перемещение.

В одной заявке перемещается только один товар. Для нескольких товаров создаются отдельные заявки.

### 20.6. OperationLog — журнал операций

Поля:

- `id`;
- `transfer_id` — необязательная ссылка на перемещение;
- `user_id`;
- `action`;
- `source_id`;
- `destination_id`;
- `product_id`;
- `quantity`;
- `status`;
- `created_at`.

Для обычного поступления `transfer_id` пустой. Для этапа перемещения он указывает на заявку.

Если перемещение будет удалено технически, связанные с ним записи журнала удалятся каскадно. Пользователь, товар и склады в журнале защищены через `PROTECT`.

## 21. Как операции меняют таблицы

| Действие | Product | Stock отправителя | Stock получателя | Transfer | OperationLog |
|---|---:|---:|---:|---:|---:|
| Создать новый товар | создаётся | — | создаётся/устанавливается начальный остаток | — | `Первичное поступление` |
| Оформить поступление | без изменений | — | `+ количество` | — | `Поступление товара` |
| Создать перемещение | без изменений | без изменений | без изменений | создаётся | `Создано перемещение` |
| Согласовать | без изменений | без изменений | без изменений | меняется статус и `approved_at` | `Перемещение согласовано` |
| Отклонить | без изменений | без изменений | без изменений | статус `rejected` | `Перемещение отклонено` |
| Отправить | без изменений | `- количество` | без изменений | статус `in_transit`, `shipped_at` | `Товар отгружен` |
| Получить | без изменений | без изменений | `+ количество` | статус `completed`, `received_at` | `Товар получен...` |
| Импорт Excel | создаётся/обновляется | устанавливается точное значение | устанавливается точное значение | — | одна общая запись импорта |

Каждая складская операция выполняется внутри транзакции базы данных. При ошибке частично выполненные изменения откатываются.

При изменении остатков во время поступления, отправки и получения используются блокировки строк базы. Это уменьшает риск неправильного количества при одновременной работе нескольких пользователей.

## 22. Важные ограничения текущей версии

- При создании перемещения товар не резервируется. Несколько заявок могут быть созданы на один и тот же остаток; окончательная проверка выполняется при отправке.
- Одна заявка содержит только один товар.
- Частичная отправка и частичное получение не поддерживаются.
- После отправки нельзя изменить принятое количество: получатель подтверждает всё количество заявки.
- Отмена заявки не реализована, хотя статус `Отменено` предусмотрен.
- Редактирование уже созданной заявки не реализовано.
- Возврат завершённого перемещения назад не реализован; требуется новое обратное перемещение.
- Отклонение через Telegram не реализовано.
- Импорт остатков заменяет текущее значение и создаёт историческую запись по каждой строке.
- Печать этикетки не меняет остаток.
- На странице номенклатуры обычный пользователь может увидеть остатки по всем складам.
- Для заявок, которые долго ожидают согласования, отправки или получения, фоновый сервис отправляет повторные Telegram-напоминания исполнителю и администраторам. Интервалы задаются переменными `TRANSFER_REMINDER_*`.

Исторические записи не зависят от последующего редактирования справочников. Перемещение и каждая операция сохраняют собственные снимки названия и категории товара, артикула, штрихкода, названий складов, пользователя и количества. В журнале также хранится остаток после складской операции, когда он применим.

## 23. Краткие рабочие инструкции по ролям

### Ответственный склада

Пришла поставка:

`Остатки → + Поступление товара → выбрать склад → выбрать товар → количество → Сохранить`

Нужно передать товар:

`Перемещения → + Новое перемещение → откуда → куда → товар → количество → Сохранить`

Заявка разрешена:

`Перемещения → открыть #номер → Подтвердить отправку`

Товар приехал:

`Перемещения → открыть #номер → Подтвердить получение`

### Согласующий

`Перемещения → статус На согласовании → открыть #номер → Согласовать или Отклонить`

### Главный администратор

Новая модель:

`Номенклатура → + Добавить товар → заполнить данные и начальный остаток → Сохранить`

Новый сотрудник:

`Управление → + Пользователь → заполнить данные → Сохранить`

Новый склад:

`Управление → + Склад → назначить ответственного и согласующего → Сохранить`

Массовая загрузка:

`Импорт Excel → выбрать .xlsx → Импортировать`
